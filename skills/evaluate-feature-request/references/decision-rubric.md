# Feature decision rubric

Score each dimension from 0 to 3. The score organizes evidence; it does not replace judgment.

| Dimension | 0 | 1 | 2 | 3 |
| --- | --- | --- | --- | --- |
| Problem evidence | No verified problem | One unclear report | Repeated concrete problem | Repeated problem with costly impact |
| User reach | Unknown or isolated | Narrow segment | Meaningful segment | Broad or strategically critical segment |
| Product fit | Conflicts with direction | Weak adjacency | Clear adjacency | Directly advances stated direction |
| Existing coverage | Already solved | Mostly solved | Partial workaround | No adequate solution |
| Roadmap fit | Conflicts or duplicates | Crowds committed work | Fits with tradeoffs | Reinforces committed work |
| Delivery cost | Disproportionate or unknown | Large cross-cutting change | Bounded multi-surface change | Small, well-understood change |
| Ongoing cost | High permanent burden | Material maintenance | Manageable maintenance | Negligible or reduces burden |
| Risk | Unbounded | Significant unresolved risk | Known mitigations | Low and reversible |

## Verdict rules

### Pursue now

Use when the problem and product fit are strong, existing coverage is inadequate, and cost and risk are understood well enough to commit. Name the roadmap tradeoff.

### Validate first

Use when the problem may matter but one decisive uncertainty remains. Recommend the smallest evidence-gathering action and define the signal that would justify pursuit.

### Defer

Use when the request is credible but timing, roadmap fit, dependencies, or cost makes it a poor current commitment. Name the condition for reconsideration.

### Decline

Use when evidence shows weak fit, adequate existing coverage, disproportionate cost, or unacceptable durable risk. Offer a supported alternative when one exists.

### Already addressed

Use when the current product or committed work satisfies the underlying outcome. Cite where it exists and identify any remaining discoverability or adoption gap.

## Decision discipline

- Separate demand volume from problem severity.
- Separate implementation effort from permanent operating cost.
- Distinguish a repository fact from an inference about strategy.
- Treat missing evidence as uncertainty, not evidence against the request.
- State the strongest case against the selected verdict before finalizing it.
